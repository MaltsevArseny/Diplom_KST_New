**Хронометраж: 10–15 минут**

## 1. Вступление (1 мин.)

Добрый день, уважаемые члены комиссии, коллеги и гости.

Меня зовут [ИМЯ], и сегодня я представляю свой дипломный проект — десктопное приложение **DigitalHub** для продажи компьютерного оборудования.

Прежде чем перейти к слайдам и техническим деталям, хочу задать вопрос: кто из вас хотя бы раз заходил в небольшой магазин техники, где продавец ищет товар в Excel-таблице или вообще в тетрадке? Мне это знакомо. И именно из этой ситуации родилась идея проекта.

## 2. Постановка задачи (1,5 мин.)

Представьте небольшой магазин компьютерных комплектующих. У него нет стабильного интернета, нет бюджета на 1С или облачную CRM. Нужна простая программа, которая:

- хранит каталог из 500 товаров прямо на компьютере;
- даёт покупателям удобный интерфейс для выбора и заказа;
- администратору — инструмент для управления всем этим.

И всё это — без серверов, без облака, без подписки. Один файл базы данных, один JAR-файл программы.

Важно подчеркнуть: задача дипломного проекта — не вывести продукт на рынок, а **продемонстрировать практическое владение технологиями**, которые мы изучали в колледже. Каталог, корзина, заказы, безопасность, админ-панель, тестирование — это полигон, на котором я отрабатывал реальные навыки разработки.

## 3. Ограничения и допущения проекта (1,5 мин.)

Любой проект работает в определённых рамках. Для дипломной работы эти рамки особенно важны: они показывают, что автор понимает границы своего решения и не выдаёт учебный проект за промышленную систему.

**Техническая среда:**
- Приложение рассчитано на **одного пользователя за компьютером** одновременно. SQLite работает в режиме одного подключения — для демонстрационного терминала в магазине это полностью оправданно.
- Минимальное разрешение экрана — **1280×720**. На практике это покрывает абсолютное большинство десктопов.
- **Автономная работа.** Вся логика — на одном компьютере, без интернета. Обратная сторона: синхронизация между точками продаж потребовала бы отдельного серверного компонента, а это уже за рамками данной работы.

**Функциональные допущения:**
- Оплата — **только при получении**. Интеграция с платёжными шлюзами требует интернета и сертификации — это выходит за рамки учебного проекта.
- Изображения товаров — **встроены в приложение**. Для дипломной работы это упрощает развёртывание; в промышленном решении их стоило бы вынести во внешнюю папку.
- Роли — строго **две**: покупатель и администратор. Для демонстрации ролевой модели этого достаточно.

**Безопасность:**
- Ключ шифрования AES-256 хранится в конфигурации приложения. В реальной коммерческой системе его следовало бы вынести в переменные окружения или аппаратный модуль. Для учебного проекта важнее продемонстрировать сам механизм шифрования — и он реализован корректно.

## 4. Выбор архитектуры — почему именно так (2,5 мин.)

Архитектурные решения — это не про «что модно», а про «что решает задачу и демонстрирует компетенции». Расскажу, как и почему я пришёл к текущему стеку.

### Почему десктоп, а не веб?

Первый вопрос, который у многих возникает: «Зачем десктоп в 2026 году?» Ответ — **требования задачи**. Целевой пользователь — магазин без стабильного интернета. Веб-приложение здесь буквально не работает. Кроме того, desktop-проект позволил мне поработать с JavaFX, горячими клавишами, управлением окнами, прямым доступом к файловой системе — то есть с теми вещами, которые в веб-разработке просто не встречаются.

### Почему Java 21 + JavaFX, а не Electron или C#/WPF?

Я рассматривал три варианта:

| Критерий | Java + JavaFX | Electron | C# + WPF |
|----------|--------------|----------|-----------|
| Кроссплатформенность | Windows, macOS, Linux | Все платформы | Только Windows |
| Размер дистрибутива | ~30 МБ (fat JAR) | ~150 МБ (Chromium внутри) | ~10 МБ, но привязка к .NET |
| Потребление RAM | ~80–120 МБ | ~300–500 МБ | ~60–100 МБ |
| Мой уровень владения | Java изучалась в колледже | JS — поверхностно | C# не изучался |

Electron отпал из-за гигантского потребления ресурсов. WPF — отличный фреймворк, но привязывает к одной ОС, а я хотел сохранить кроссплатформенность. Java + JavaFX — осознанный выбор: я владею языком, дистрибутив компактный, а работа с пройденным на занятиях стеком позволяет сосредоточиться на архитектуре, а не на освоении незнакомого инструмента.

### Почему MVVM и слоистая архитектура?

Приложение разделено на пять слоёв:

```
View (18 классов)  →  Service (7 классов)  →  Repository (10 классов)  →  SQLite (3НФ)
         ↕                                          ↑
    EventBus ←──────── Config (5 классов) ──────────→
```

- **Model** — чистые объекты данных: `User`, `Product`, `Order`, `CartItem`. Покрыты тестами на 98%.
- **Repository** — интерфейс + реализация для каждой сущности (`IProductRepository` → `ProductRepository`). Вся работа с SQL изолирована.
- **Service** — бизнес-правила: проверка корзины, блокировка аккаунта, оформление заказа. Тестируются без запуска интерфейса.
- **View** — 18 JavaFX-экранов.
- **Config** — `EventBus`, `SessionManager`, `AppConfig`, `DatabaseManager`, `SeedProducts`.

Зачем столько слоёв для дипломного проекта? Именно ради демонстрации: я показываю, что умею проектировать систему так, чтобы каждый компонент можно было тестировать и заменять независимо. 235 тестов — прямое следствие этой архитектуры.

### Почему SQLite, а не H2 или Derby?

- **H2** — мощнее, но создаёт несколько файлов и избыточна для 9 таблиц.
- **Derby** — менее популярна, документации меньше.
- **SQLite** — один файл, 0 конфигурации, огромное сообщество. Её используют Firefox, Chrome, Android — если SQLite достаточна для браузера с миллиардом пользователей, для учебного каталога из 500 товаров она справится.

### EventBus — зачем своя шина событий?

Когда покупатель добавляет товар в корзину, бейдж на иконке должен обновиться. Без шины событий пришлось бы передавать ссылку на `MainLayout` во все экраны. `EventBus` развязывает компоненты: каталог публикует `CART_UPDATED`, навбар подписан на это событие. Паттерн Observer — один из паттернов, которые я хотел продемонстрировать.

## 7. Безопасность (1,5 мин.)

**Хеширование паролей.** Каждый пароль хешируется с уникальной 128-битной солью. Даже два пользователя с одинаковым паролем получат разные хеши. Сравнение выполняется за *постоянное время* — это защита от timing-атак.

**Защита от перебора.** Пять неудачных попыток — аккаунт блокируется на 5 минут. Параметры вынесены в конфигурацию.

**Шифрование PII.** Телефоны и адреса зашифрованы AES-256 с уникальным IV для каждой записи. Даже при физическом доступе к файлу БД — набор символов, а не реальные данные.

Всю эту систему я реализовал не ради формальных требований. Мне было важно показать, что я понимаю: работа с данными предполагает ответственность. Даже в рамках учебного проекта стоит делать так, как правильно.

## 8. Тестирование (1,5 мин.)

Тесты — не «обязательная формальность», а инструмент, который ловит баги до того, как их найдёт пользователь. И один из ключевых навыков, который я хотел продемонстрировать в этом проекте.

**235 тестов, 26 тестовых классов.** Запуск одной командой `mvn test` — все проходят.

**Покрытие кода** (JaCoCo):
- Модели данных: 98%
- Конфигурация: 94%
- Модуль безопасности: 87%
- Общее покрытие (без UI): 84% инструкций, 55% ветвлений

UI не тестируется автоматически — для JavaFX нужен TestFX и headless-режим. Зато вся бизнес-логика покрыта: валидация заказа, блокировка аккаунта, шифрование, работа с базой.

## 9. Что я вынес из этого проекта (1 мин.)

Несколько уроков, которые не прочитаешь в учебнике:

1. **SQLite может заблокировать файл.** Два соединения одновременно — и запись ломается. Пришлось перестроить `DatabaseManager`: каждая операция — атомарная, соединение живёт миллисекунды. Мелочь, но стоила полдня дебага.

2. **Тесты экономят время.** Когда я рефакторил репозитории (добавлял интерфейсы), тесты за секунду показали, где я что-то сломал. Без них — полчаса ручного тестирования.

3. **Архитектура — это инвестиция.** Разделение на слои казалось избыточным на старте. Но когда проект вырос до 18 экранов и 52 классов — именно архитектура позволила не утонуть в зависимостях.

## 10. Планы на развитие — что можно улучшить (1,5 мин.)

Дипломный проект — это финишная прямая обучения, но не финал развития. Вот что можно было бы сделать дальше, если продолжать работу над DigitalHub:

**Сетевой режим.** Сейчас данные живут в одном файле. Логичное развитие — добавить лёгкий REST-API, чтобы несколько терминалов работали с общей базой. Благодаря интерфейсам репозиториев (`IProductRepository`, `IOrderRepository`) переход на PostgreSQL или MySQL потребует переписать только реализации, не трогая бизнес-логику.

**Внешние изображения товаров.** Сейчас картинки зашиты в JAR. Перенос их во внешнюю папку позволит администратору загружать фото новых товаров через интерфейс — без пересборки.

**UI-тесты.** 235 тестов покрывают логику, но интерфейс проверяется только вручную. **TestFX** позволяет запускать JavaFX в headless-режиме и автоматизировать проверку переходов, нажатий, ввода данных.

**Система уведомлений.** Сейчас администратор узнаёт о новых заказах, только зайдя в соответствующий раздел. Всплывающие уведомления при поступлении заказа — простая, но полезная доработка.

**Расширение ролевой модели.** Две роли для демонстрации достаточно, но для реального применения полезно ввести «менеджера» с ограниченным доступом и гибкую систему прав.

## 12. Ожидаемые вопросы комиссии и ответы

### «Почему не веб-приложение? Десктоп — это устаревший подход.»

Десктоп — не устаревший, а *целевой* подход для данной задачи. В условиях задания — отсутствие стабильного интернета. Веб-приложение здесь просто не работает. Кроме того, desktop-проект позволил мне освоить вещи, которых в веб-разработке нет: управление окнами, системные горячие клавиши, прямую работу с файловой системой. Инструменты вроде IntelliJ IDEA, VS Code, 1С — тоже десктопные приложения, и их никто не называет устаревшими.

### «Как вы решаете проблему одновременной работы нескольких пользователей?»

В текущей реализации — один пользователь за компьютером. Для дипломной работы это осознанное ограничение: задача — продемонстрировать навыки, а не построить распределённую систему. При этом архитектура подготовлена к расширению: интерфейсы репозиториев позволяют заменить SQLite на сетевую СУБД, не переписывая бизнес-логику.

### «Зачем шифрование в учебном проекте?»

Потому что учебный проект — лучшее место, чтобы научиться делать правильно. В базе хранятся телефоны и адреса. Я мог бы оставить их открытым текстом, но тогда я не освоил бы AES-256, не узнал бы про IV и режимы шифрования, не написал бы тесты на это. Дипломная работа — это инвестиция в навыки, и мне было важно показать, что я умею работать с криптографией.

### «Почему не использовали ORM (Hibernate, MyBatis)?»

Осознанный выбор. ORM генерирует SQL за вас, а я хотел написать его руками — чтобы показать, что я понимаю, что происходит: `JOIN`, `GROUP BY`, транзакции, `PreparedStatement` для защиты от инъекций. Кроме того, для 9 таблиц ORM — лишняя зависимость, которая утяжеляет fat JAR и скрывает логику.

### «Почему покрытие ветвлений только 55%?»

JaCoCo считает ветвления строго: каждый `if`, `switch`, `try-catch` — это минимум две ветки. Многие из незакрытых ветвей — обработка исключений (`catch`-блоки), которые сложно вызвать без mock-фреймворка (например, ошибка шифрования, обрыв JDBC-соединения). При этом 84% покрытие инструкций — это высокий показатель, особенно для проекта без Mockito.

### «Какие паттерны проектирования вы использовали?»

1. **MVVM** — разделение на Model, View, Service/ViewModel. Позволяет тестировать логику без интерфейса.
2. **Repository** — абстракция доступа к данным через интерфейсы. Делает замену СУБД безболезненной.
3. **Singleton** — `DatabaseManager`, `SessionManager`. Единая точка входа к базе и к сессии.
4. **Observer** — `EventBus` рассылает события, подписчики реагируют без жёстких связей.
5. **Lazy Loading** — каталог подгружает товары порциями по 15 штук.

### «Что было самым сложным?»

Конкурентный доступ к SQLite. Когда два участка кода пытаются одновременно записать, SQLite выбрасывает `SQLITE_BUSY`. Я полностью переработал `DatabaseManager`: каждый блок «открыл → прочитал/записал → закрыл» стал атомарным. Это научило меня думать о жизненном цикле ресурсов — навык, который пригодится в любом серверном приложении.

### «Как происходит развёртывание?»

На целевом компьютере нужна Java 21 или выше. Далее — копируем папку `dist/` с JAR-файлом и запускаем `run.bat`. При первом запуске приложение само создаёт базу, заполняет каталог и генерирует тестовые данные. Никаких инсталляторов, никаких зависимостей кроме JRE.

### «Как вы тестировали безопасность?»

Модуль `SecurityManager` покрыт тестами на 87%. Проверяется:
- цикл «хеширование → верификация» для корректного и некорректного пароля;
- уникальность солей: два хеша одного пароля — разные;
- шифрование/дешифрование PII: запись → шифрование → дешифрование → проверка;
- блокировка после 5 неудачных попыток → автоматическая разблокировка через 5 минут.

Ручное тестирование: я открывал файл БД через DB Browser for SQLite и проверял, что вместо телефонов и адресов — зашифрованные строки.

### «Почему именно Java 21?»

Java 21 — это актуальная LTS-версия (Long Term Support), поддерживаемая до 2031+ года. Выбор LTS-версии для дипломного проекта — осознанное решение: проект стабилен, зависимости совместимы, и при необходимости его можно запустить и через год без обновления стека. Java 21 привносит улучшения производительности, паттерн-матчинг, виртуальные потоки и другие нововведения, делая код современным и поддерживаемым.
